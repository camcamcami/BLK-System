# BLK-SYSTEM-010 — Fixture-to-Live BLK-test MCP Gap Register

**Status:** Task 2 review artifact  
**Sprint:** `blk-system-010`  
**Purpose:** Document the exact gaps between current fixture-only BLK-test behavior and any future live BLK-test MCP path without authorizing live transport, BEO publication, RTM generation, tactical LLM execution, cyber execution, or active-vault access.

---

## 1. Scope and non-authority statement

This register compares current disabled/fixture behavior from BLK-013, BLK-014, BLK-015, and BLK-016 against BLK-008's future target-state MCP physics-oracle design.

Sprint 010 does not authorize live BLK-test MCP. It does not start a live MCP server/client, call live MCP transport, run Codex, call live tactical LLMs, run cyber tooling, publish authoritative BEOs, generate RTM, grant RTM drift rejection authority, read protected BLK-req vault bodies, or claim production sandbox/secret isolation.

Each gap row separates current fixture evidence from future authority. A row marked as a future target is not executable authority.

## 3. Future implementation slice recommendations

Future implementation slice recommendations, in safe order:

1. disabled-by-default MCP transport lifecycle proof;
2. fixed-tool registry and no arbitrary shell contract;
3. workspace clone/isolation, teardown, locking, and parallel execution prevention proof;
4. process tree kill/timeout/flood behavior plus Output compression gates;
5. source-bound live request contract and PASS/FAIL/BLOCKED mapping preservation;
6. BLK-test-specific approval-channel mechanics;
7. Secret/network isolation policy and active-vault file-access prohibition;
8. audit logging and replay evidence bundle;
9. BEO publication design only after live BLK-test evidence is source-bound and approved;
10. RTM ledger design only after BEO publication authority is separately settled.

No item above is authorized by this register. Every item requires a later sprint with its own RED/GREEN gates, explicit non-goals, and human approval boundary.

## 4. Acceptance criteria for this register

This register is acceptable only if it preserves these boundaries:

- live BLK-test MCP remains disabled;
- `blk-test` consumes source evidence and does not mutate source or replace `blk-pipe`;
- PASS/FAIL require source-bound canonical trace evidence and non-empty checks;
- BLOCKED remains not-run/non-authoritative evidence;
- BEO draft-only boundary and RTM non-generation remain explicit;
- active BLK-req vault bodies remain unread in Sprint 010;
- future work is decomposed into narrow slices rather than bundled live authority.
